# ClodRail / RailwaySystem 项目整体文档

## 1. 项目简介

本项目是一个高仿 **12306 铁路购票系统** 的前后端分离 + 微服务实战项目，覆盖从用户登录、车票查询、下单支付、订单状态流转、积分商城、到 AI 客服实时会话等完整业务链路。

项目由两个前端应用与多个后端服务组成，所有 HTTP API 统一通过网关对外提供访问；部分实时通信能力通过 Netty WebSocket 独立端口对外提供。

## 2. 技术栈总览

## 2.1 后端技术栈

- **语言/构建**

  - Java（父工程声明 `java.version=17`，各模块 `maven.compiler.source/target` 在 21/24 不等，需以实际编译环境为准）
  - Maven 多模块工程

- **核心框架**

  - Spring Boot **3.5.5**
  - Spring Cloud **2025.0.0**
  - Spring Cloud Alibaba **2023.0.3.2**

- **微服务基础设施**

  - Nacos：服务注册发现 + 配置中心
  - Spring Cloud Gateway（WebFlux）：统一入口、路由、鉴权
  - OpenFeign + LoadBalancer：服务间调用

- **存储与中间件**

  - MySQL 8+：业务主库
  - Redis 7+：缓存、原子扣减、冲突控制、Token 会话
  - RabbitMQ：消息解耦、异步任务
  - Seata：分布式事务（工程里既出现 AT 配置，也有 TCC 接口骨架）
  - Elasticsearch：商城检索能力（工程封装了 ES Template）
  - Netty：WebSocket 长连接（AI/客服会话）

- **常用库/组件**
  - MyBatis（`mybatis-spring-boot-starter`）
  - Druid：连接池
  - PageHelper：分页
  - Lombok、Hutool
  - Knife4j + Springdoc：OpenAPI 文档
  - LangChain4j：大模型集成
  - XXL-JOB：分布式任务调度

## 6. 关键横切能力设计

## 6.1 认证（JWT + Redis 会话）

- 前端请求在 `axios` 请求拦截器中设置：

  - 用户端：`Authorization: Bearer <token>`（localStorage `token`）
  - 管理端：`Authorization: Bearer <token>`（localStorage `admin_token`）

- 网关侧鉴权：`rs-gateway/src/main/java/com/rs/filters/SecurityFilter.java`
  - 若请求命中放行白名单（`ExcludeProperties`）则直接放行
  - 解析 `Authorization` Header（工程常量名为 `AUTHENTICATION`，前缀 `Bearer `）
  - 使用 Header 中携带的 `uuid` 拼接 Redis Key：
    - 用户：`user:login:token:<uuid>`
    - 管理员：`admin:login:token:<uuid>`
  - Redis 中取回 JWT，使用 `JWTUtil.parseJWT()` 解析 `subject` 并反序列化为 `User/Admin`
  - 将 `userId/adminId` 写入 `exchange.getAttributes()`（供后续链路使用）

> 现状观察：`JWTUtil` 中 `JWT_KEY` 为硬编码常量。生产环境建议迁移为配置化（环境变量/Nacos），并定期轮换。

## 6.2 管理端权限（RBAC）

- 网关对管理端请求（`/admin/**`）进行权限校验
- 通过 Redis Set 保存“角色允许访问的 API 列表”，并按 `METHOD-pathPattern` 匹配
- 若管理员角色等于特定值（代码里示例 `role == 104`）则视为超级管理员直接放行

## 6.3 服务间调用（OpenFeign）

- `rs-api` 模块中集中定义 Feign Client 与 DTO
- 网关配置中也存在 `spring.cloud.openfeign.client.config` 超时配置示例
- 各业务服务依赖 `rs-api` 从而统一跨服务接口契约

## 7. 核心业务模块说明

## 7.1 用户服务（rs-user / user-service）

**主要职责**

- 用户注册/登录/认证
- 乘车人/联系人信息维护（`/customer/contact/**`）
- 管理员登录与管理端用户管理（`/admin/**`）
- 权限与角色相关的内部接口（`/inner/**`）

**数据库（已提供 SQL）**

- `rs-user/src/main/resources/sql/create.sql`：`user` 表（含 username/email/phone/id_card 唯一约束）

## 7.2 票务服务（rs-ticket / ticket-service）

**主要职责**

- 车站、线路、车次、车票管理
- 余票查询与座位库存
- 管理端：车站/线路/车次/车票的 CRUD

**数据库（已提供 SQL）**

`rs-ticket/src/main/resources/sql/create.sql` 包含：

- `station`：车站
- `line`：线路
- `line_stations`：线路途经站
- `train`：车次/列车
- `ticket`：车票（start/end、价格、是否热门等）
- `seat`：座位明细（含 `full_seat_code` 生成列）
- `stop_stations`：停靠站与到离时间

并包含：

- `auto_create_seat_by_ticket` 触发器：新建 ticket 后自动生成座位
- `create_seat` 存储过程：批量写入 seat

**热门票/缓存预热（XXL-JOB）**

`HotTicketJobHandler` 体现两类任务：

- `ticketRefreshTodayHotTickets`：维护当天热门 ticketId Set：`ticket:hot:yyyy:MM:dd`
- `ticketPreloadHotTicketStock`：提前 15 分钟预热即将开售的热门票库存：`ticket:store:<ticketId>:<seatType>`

## 7.3 订单服务（rs-order / order-service）

**主要职责**

- 下单、支付、订单查询
- 订单超时/状态流转
- 订单相关消息生产/消费
- 与票务服务交互（占座/释放座位）

**分布式事务（Seata）**

- `bootstrap.yaml` 中引入：`shared-seata.yaml` 并配置 `seata.tx-service-group`
- 工程内存在 `CreateOrderTCCTranslation` 接口（`@LocalTCC`），当前为空壳，表示 TCC 方向的扩展点
- 同时 `bootstrap.yaml` 中出现 `data-source-proxy-mode: AT`，说明现阶段更偏向 AT 模式；是否启用 TCC 需以业务代码为准

**订单超时（XXL-JOB + Redis Lua）**

`OrderExpireJobHandler`：

- 扫描“过期未支付订单”并置为过期状态
- 调用票务/座位服务回滚占座
- 使用 Redis Lua 脚本回滚“乘客时间冲突”位图/标记（跨天/非跨天两套脚本）

## 7.4 积分商城（rs-mall / mall-service）

**主要职责**

- 用户积分管理
- 商品展示/搜索（ES）
- 积分兑换下单（商城订单）
- 管理端：商品与订单管理

**ES 索引同步（RabbitMQ + Canal 消息）**

- `ItemCanalListener` 监听 `canal.item.queue`
- 消息类型 `INSERT/UPDATE/DELETE` 对应 ES 批量写入/更新/删除

**数据库（已提供 SQL）**

- `rs-mall/src/main/resources/sql/mall_order.sql`：`mall_order` 表

## 7.5 AI/客服助手（rs-assistant / assistant-service）

**主要职责**

- HTTP API：用户端 AI 问答、客服相关接口（`/customer/assistant/**`）
- 管理端：会话记忆/历史等（`/admin/memory/**`）
- Netty WebSocket：实时会话（用户/客服两类通道）

**消息驱动的会话记忆（RabbitMQ）**

`MsgListener`：

- 监听 `rs.assistant.msg` Exchange 下不同 routing key：
  - `assistant.user`
  - `assistant.order`
  - `assistant.agent`
- 将会话消息落库到 `Memory`（并结合 Redis 补充订单信息）

**Netty WebSocket**

- `NettyConfig` 在 `@PostConstruct` 启动独立线程启动 Netty Server（默认端口 `18085`）
- `WebSocketHandler` 按 URI 区分：
  - `/ws/assistant/user...`：用户端 WS
  - `/ws/assistant/agent...`：客服端 WS

## 8. 典型端到端业务流程（文字版）

## 8.1 “搜索-下单-支付”的主链路

- **查询**：

  - 用户端调用 `/customer/tickets/**` `/customer/stations/**` 等
  - 网关路由到 `ticket-service`
  - 票务服务从 MySQL/Redis 组合查询并返回

- **下单**：

  - 用户端调用 `/customer/orders/**`
  - 网关路由到 `order-service`
  - 订单服务校验余票/占座（通过 Feign 调用票务相关内部接口）
  - 写入订单（热门票会走 Redis 预扣、异步落库等策略，详见 `Docs/ClodRail项目介绍.md`）

- **支付**：

  - 用户端调用 `/customer/pay/**`
  - 订单服务对接第三方支付（工程中存在支付宝相关依赖与配置导入）

- **异步处理**：
  - 下单成功后，订单服务通过 RabbitMQ 发送消息
  - 消费端进行订单后置动作（如积分发放、异步落库/状态推进等）

## 8.2 积分发放

- `order-service` 消费 `order.point`（见 `TicketOrderListener.ticketOrderPointListener`）
- 根据订单金额组装 `AddPointReqDTO`
- 通过 Feign 调用 `mall-service` 的积分接口完成入账

## 8.3 商城检索与索引同步

- `mall-service` 提供 ES 查询能力（`rs-es` 封装 `ElasticSearchTemplate`）
- Canal 将 MySQL 表变更转为 MQ 消息
- `ItemCanalListener` 消费消息并同步 ES 索引

## 8.4 客服实时会话

- 管理端 Vite 代理 `/ws` 到 Netty `18085`，建立 WebSocket
- WS 握手后按 URI 分发到用户/客服 handler
- 消息可落入 RabbitMQ 进行异步持久化（会话记忆）
